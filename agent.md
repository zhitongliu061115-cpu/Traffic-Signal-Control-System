# Agent 开发协作规范

本文件是本仓库的软件开发与 AI 编程协作规范。适用于人工开发者、AI Agent 和自动化工具。

目标：让每次修改有计划、有边界、有验证、有文档同步，并让下一位成员能从公共信息快速接手。

## 1. 规范优先级

发生冲突时按以下顺序执行：

1. 当前任务的明确需求和安全要求。
2. 本文件 `agent.md`。
3. 模块权威文档和接口协议。
4. 对应模块现有代码、测试和局部约定。

`rules.md` 只作为兼容入口，不再维护另一套重复规则。

## 2. 项目事实与强制架构

当前技术栈：

- `sys-frontend/`：Vue 3、TypeScript、Vite、Pinia、Three.js、MapLibre。
- `backend/`：Java 17、Spring Boot、WebSocket、JDBC/JPA、Flyway。
- `sim-python/`：独立 Python CityFlow 仿真服务。
- `cloud/traffic-r/`：独立 Traffic-R 模型服务。

固定调用边界：

```text
Vue 前端
  -> Spring Boot REST / WebSocket
  -> Python CityFlow 服务
  -> CityFlow Engine

Spring Boot
  -> Traffic-R /predict-batch
```

必须遵守：

- 前端只访问 Spring Boot，不直连 CityFlow、Traffic-R 或数据库。
- Spring Boot 是协议、会话、策略调度、权限、审计和持久化边界。
- Python CityFlow 只负责仿真运行、状态快照和控制动作应用。
- Traffic-R 是主要 RL 方向；MaxPressure 只作为基线和故障回退。
- 前端信号状态只以 `sim.frame.data.signals` 为准，不能用模型意图伪造真实灯色。

## 3. 单一公共工作台账

跨文件、改代码或影响他人的任务开始前，必须先在 `docs/WORK_TRACKER.md` 建立或更新任务项，再修改代码。

每个任务项至少包含：

- 唯一编号、优先级、负责人和状态。
- 目标、修改范围和明确验收标准。
- 关联风险、接口或部署影响。

状态只使用：`待开始`、`进行中`、`阻塞`、`待验证`。

以下内容不需要建立任务项：

- 纯阅读、解释或方案讨论。
- 不改变语义的单处错别字修正。

规则：

- 开始编码前把任务改为 `进行中`。
- 遇到阻塞时记录阻塞原因、已尝试方法和解除条件，不能只写“有问题”。
- 代码完成但验证未完成时使用 `待验证`，不能提前标为完成。
- 验收完成后，先把长期有效的信息同步到对应权威文档，再从台账删除完成项。
- 已完成工作的历史由 Git 提交和 PR 保存，工作台账不长期堆积完成项。

## 4. 开发前流程

每项开发任务按以下顺序执行：

1. 阅读 `docs/WORK_TRACKER.md`、相关模块文档和现有实现。
2. 检查 `git status --short`，识别并保护他人的未提交修改。
3. 明确需求边界、调用链、数据来源和验收标准。
4. 在工作台账记录 2 至 5 个实施步骤、影响文件和验证方式。
5. 确认没有重复实现或已有公共工具后再编码。

需求不清但可以从代码、测试或文档确认时，先自行查证；只有关键业务选择无法判断时才向任务提出者确认。

## 5. 修改原则

- 先读后改，优先沿用现有目录、类型、命名和工具。
- 一次任务只解决一个清晰目标，不混入无关重构。
- 修改范围尽量小，但必须完成端到端行为，不能只做表面占位。
- 不用 mock、硬编码成功值或吞异常来掩盖真实失败。
- 不新增无必要依赖；新增依赖必须说明用途、替代方案和部署影响。
- 复杂逻辑放在领域服务或复用模块中，Controller 和页面只做协议入口与组合。
- 发现半成品、漏洞或暂时无法修复的问题时，必须记录，不能只留在聊天或个人笔记中。

## 6. 编码规范

### 6.1 通用

- 名称表达业务含义，避免 `data1`、`temp2`、`handleThing` 等模糊命名。
- 函数保持单一职责；只有真实复用或降低复杂度时才增加抽象。
- 注释解释设计原因、约束或非显然行为，不复述代码。
- 错误信息应包含可定位上下文，但不得暴露密钥、令牌和内部堆栈。
- 配置通过环境变量或配置文件注入，不硬编码账号、密钥、私有路径和环境专属地址。

### 6.2 前端

- API 请求统一放在 `sys-frontend/src/api/`。
- 共享类型放在 `sys-frontend/src/types/`，跨组件状态放在 `stores/`。
- WebSocket、订阅和可复用副作用放在 `composables/`。
- 必须处理加载、空数据、错误、断线重连和真实数据不可用状态。
- 演示数据与真实数据必须有明确来源标记，不能静默混合后伪装成真实指标。
- 控制按钮必须调用真实后端能力，尚未接通时应禁用或明确标识，不能只修改本地 UI 制造成功效果。

### 6.3 Spring Boot 后端

- Controller 只负责校验、协议转换和调用 Service。
- 跨服务调用通过明确的 Client 边界，业务编排放在 Service。
- REST 使用 `/api/v1` 前缀和项目现有 `ApiResponse` 信封。
- DTO 字段、错误码和 WebSocket 消息必须与接口文档一致。
- 异步策略、会话和资源必须有超时、释放、失败隔离和可观察日志。
- 数据库结构变更只能通过新增 Flyway migration 演进，不改写已共享 migration。

### 6.4 Python CityFlow 与 Traffic-R

- `sim-python` 不承载前端协议、用户权限或页面业务。
- mock 与真实 CityFlow 模式必须可区分，验证真实链路时检查 `/health.engineMode`。
- 控制动作必须按 `sid` 隔离，会话停止和自然结束都要释放 worker、Engine 和应急状态。
- Traffic-R 正式调度使用批量接口，不能按高频仿真帧逐路口同步调用。
- 模型输出必须校验来源、相位、完整性和解析状态；失败按现有回退策略处理。

## 7. 文档管理与同步矩阵

优先更新已有权威文档。只有现有文档无法承载一个长期独立主题时才能新建文档，并同时更新对应 `README.md` 索引。

同一事实只在一个权威文档详细描述，其他文档使用链接和简短摘要，禁止复制大段内容。

| 变更类型 | 必须检查并更新的文档 |
|---|---|
| 当前任务、未完成工作、跨组阻塞 | `docs/WORK_TRACKER.md` |
| 前后端接口、DTO、WebSocket、认证头 | `backend/docs/API_GUIDELINES.md`；跨前后端协议同时更新 `docs/CFRP-1.0-前后端通信协议.md` |
| Spring Boot、CityFlow、Traffic-R 调用边界 | `backend/docs/CALL_CHAIN.md`、`backend/docs/BACKEND_ARCHITECTURE.md` |
| 包结构、模块职责、目录变化 | `docs/PROJECT_STRUCTURE.md` |
| 数据库表、字段、迁移或访问方式 | `backend/docs/DATABASE_STRUCTURE.md`、`backend/docs/TECHNICAL_DESIGN.md` |
| 服务地址、环境变量、启动或部署拓扑 | `backend/docs/DEPLOYMENT.md` 或对应 runbook |
| 半成品、漏洞、性能或兼容性风险 | `backend/docs/RISK_TODO.md`；台账只引用风险编号，不复制详情 |
| 项目定位、已实现能力和阶段变化 | `README.md`、`docs/项目简介.md` |

禁止为一次任务默认创建“总结”“记录”“最终版”“新最终版”等临时 Markdown。临时分析放在任务沟通中；需要长期保留的结论写回对应权威文档。

## 8. 问题与风险记录

发现问题时按以下方式处理：

1. 能在当前任务范围内修复：修复并补验证，不新增长期风险项。
2. 无法立即修复但影响当前任务：在工作台账标为 `阻塞`。
3. 跨任务持续存在的技术风险：在 `backend/docs/RISK_TODO.md` 建立唯一风险项，台账引用其编号。
4. 问题解决后更新风险状态；确认不再需要跟踪时清理已完成内容，避免清单无限增长。

风险记录必须说明：影响范围、严重级别、触发条件、当前状态、临时措施和最终处理方向。

## 9. 测试与验证

验证强度与变更风险匹配：

| 修改范围 | 最低验证要求 |
|---|---|
| 前端类型或逻辑 | `npm run type-check`、相关 Vitest |
| 前端构建或依赖 | `npm run build` |
| Spring Boot 业务 | 相关测试；共享链路执行 `mvn test` |
| Python CityFlow | `python -m unittest discover tests` |
| Traffic-R 接口 | 接口契约测试；真实模型结果需单独注明环境 |
| 接口或跨服务链路 | 两端契约测试或最小联调 |
| 文档 | 检查路径、链接、命令、字段和当前代码一致性 |

不能运行的验证必须写明原因、风险和后续验证人，禁止用“应该可以”代替结果。

完成前必须检查：

```powershell
git status --short
git diff --check
git diff
```

## 10. Git 与团队沟通

- 遵守 `docs/GIT_规范.md`，一个分支和一个 PR 只处理一个任务。
- 未经明确要求不自动提交、推送或修改共享分支历史。
- 提交前确认不包含 `.env`、密钥、日志、构建产物、模型权重和无关改动。
- PR 或交接说明必须包含：目标、关键修改、接口/部署影响、验证结果、未完成项和回滚关注点。
- 对后续成员有影响的决定不能只写在提交信息或聊天中，必须写回权威文档或工作台账。

## 11. 完成定义

只有同时满足以下条件，任务才算完成：

- 验收标准对应的行为已实现，不是占位或仅 UI 演示。
- 相关测试通过，或已明确记录无法验证的原因和风险。
- 接口、部署、数据库、架构等受影响文档已同步。
- 新发现且未解决的问题已进入正确的公共记录。
- 工作台账中的长期信息已迁移到权威文档，完成项已清理。
- `git diff` 只包含本任务相关修改，没有敏感信息和无关文件。

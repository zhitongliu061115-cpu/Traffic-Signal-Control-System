# 项目简介

## 项目名称

信号灯配时控制与应急通行信控系统

## 项目定位

本项目面向城市路网信号控制与交通仿真场景，目标是构建一个可演示、可联调、可扩展的交通数字孪生系统。系统以 Vue 前端大屏呈现路网、车辆、信号相位、拥堵状态、应急通行和指标对比，以 Spring Boot 作为统一业务后端，以独立 Python 服务承载 CityFlow 路网解析、仿真推进和后续算法计算。

当前主线是先打通“路网读取、仿真会话、实时帧推送、前端可视化”的闭环，再逐步接入信号控制策略、历史指标入库、应急通行任务和智能体辅助决策。

## 要解决的问题

1. 固定配时信号灯难以适应实时车流变化，高峰、事故或局部拥堵时容易产生排队扩散。
2. 多路口之间缺少协同控制，单点优化难以提升干线路网整体通行效率。
3. 救护车、消防车等应急车辆缺少沿线信号联动，通行效率和可控性不足。
4. 仿真、策略、可视化和后续审计数据容易耦合，需要清晰的服务边界和统一协议。

## 总体架构

```text
Vue + TypeScript 前端大屏
  -> Spring Boot REST / WebSocket 主后端
  -> Python CityFlow 仿真服务
  -> CityFlow Engine / mock engine / 后续算法策略
```

前端只访问 Spring Boot 主后端，不直接连接 Python 仿真服务。Spring Boot 负责 REST、WebSocket、会话管理、策略调度、数据库边界和审计扩展；Python 服务负责路网解析、仿真帧生成、真实 CityFlow 接入和控制动作应用。

## 子系统分工

| 子系统 | 主要职责 | 当前状态 |
|---|---|---|
| `sys-frontend` | 交通态势大屏、路网/车辆/信号渲染、应急通行交互、指标对比、智能体面板 | 主大屏以 Pinia mock 数据驱动，具备交通主题演示能力；`/data-analysis` 是独立数据分析演示页 |
| `backend` | REST / WebSocket 统一入口、仿真会话、帧轮询、策略调度、数据库配置 | 已提供场景路网、仿真创建/启动/暂停/停止、`sim.frame` 推送和策略回写入口 |
| `sim-python` | CityFlow 路网解析、仿真会话、实时 frame、控制动作接收 | 默认 mock engine 可运行；`SIM_ENGINE_MODE=cityflow` 预留真实 CityFlow 引擎模式 |
| `docs` | 项目结构、通信协议、后端设计、协作规范、算法资料索引 | 已包含项目结构、CFRP 协议、后端设计和 Traffic-R1 资料 |

## 当前已实现

- 前端主页面 `/`：包含交通统计、实时告警、中央路网、AI 信号控制、应急通行、效果对比和智能体问答等模块。
- 前端状态层：`sys-frontend/src/stores/traffic.ts` 使用 mock 路网、车辆、路口、告警和指标数据，按 200ms / 2s / 5s 节奏刷新演示状态。
- 后端场景接口：`GET /api/v1/scenes/{sceneId}/roadnet` 通过 `CityFlowClient` 获取 Python 服务解析后的路网。
- 后端仿真接口：`POST /api/v1/simulations`、`POST /api/v1/simulations/{sid}/start|pause|stop` 管理仿真会话。
- 后端实时推送：`SimulationFrameScheduler` 按 `cityflow.frame-poll-interval-ms` 拉取 Python frame，并通过 `/ws/v1/simulations/{sid}` 推送 CFRP `sim.frame` 消息。
- 后端策略入口：`fixed-time`、`max-pressure`、`traffic-r` 控制器已注册，`rl` 会映射到 `traffic-r`；当前 Traffic-R 外部模型尚未接入，会保持当前相位作为占位决策。
- Python 服务：提供 `/health`、`/cityflow/scenes/{sceneId}/roadnet`、`/cityflow/simulations`、`/cityflow/simulations/{sid}/frame`、`/cityflow/simulations/{sid}/actions`。
- Python 测试：覆盖路网解析、仿真帧、错误响应、并发帧序号、控制动作校验和真实 CityFlow 配置入口。

## 当前边界

- 前端交通大屏仍以 mock 数据为主，尚未接入后端 REST / WebSocket 的真实仿真帧。
- 数据库已有 H2 默认配置、PostgreSQL profile 和初始化迁移脚本，但仿真会话与指标快照的完整持久化仍待补齐。
- `fixed-time` 和 `max-pressure` 已有工程化 baseline 决策；`traffic-r` 目前是外部模型未连接时的安全占位。
- 应急通行和智能体能力在前端可演示，后端包结构与协议边界已预留，真实任务编排、路径规划、RAG / LLM 服务仍待接入。
- 真实 CityFlow 需要在具备 CityFlow 依赖的 Python 环境中运行；Windows 普通 Python 默认使用 mock engine。

## 技术栈

- 前端：Vue 3、TypeScript、Vite、Pinia、Element Plus、ECharts、MapLibre GL、Three.js。
- 后端：Java 17、Spring Boot 3.3、Spring Web、Spring WebSocket、Spring Data JPA、Flyway、H2、PostgreSQL。
- 仿真服务：Python 标准库 HTTP 服务、CityFlow 路网/流量文件、可选真实 CityFlow Engine。
- 测试：Vitest、Playwright、Spring Boot Test、Python `unittest`。

## 运行方式

后端：

```sh
cd backend
mvn spring-boot:run
```

前端：

```sh
cd sys-frontend
npm install
npm run dev
```

Python 仿真服务：

```sh
cd sim-python
python app/server.py --host 127.0.0.1 --port 9000
```

默认场景：

```text
sceneId: jinan_3x4
roadnet: sim-python/data/jinan_3x4/roadnet_3_4.json
flow: sim-python/data/jinan_3x4/flow_3_4_jinan_real.json
```

## 后续迭代重点

1. 前端接入真实 REST / WebSocket 数据，逐步替换 mock 交通状态。
2. 持久化仿真会话、指标快照和关键控制动作，形成可复盘数据链。
3. 完善策略效果对比，稳定 `fixed-time`、`max-pressure` 和 Traffic-R / RL 接入方式。
4. 将应急通行从前端演示升级为后端任务、路径规划、ETA 预测和沿线相位优先。
5. 接入智能体/RAG 服务，用于状态查询、拥堵解释、调度建议和报告生成。

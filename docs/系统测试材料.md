## 6 系统测试

系统测试围绕交通信号控制系统的核心业务链路展开，重点验证系统是否能够完成路网仿真、实时状态展示、策略调度、安全约束、数据库记录、智能体工具调用与知识库问答等功能。测试对象包括后端 Spring Boot 服务、CityFlow 仿真服务、前端可视化页面、PostgreSQL 数据库、Traffic-R 模型服务、Agent 编排模块和百炼知识库检索接口。

本系统测试采用“单元测试 + 接口测试 + 脚本化联调测试 + 人工页面验证”的方式进行。其中后端核心服务使用 JUnit 进行单元测试；Agent 工具调用使用 PowerShell 脚本批量验证；仿真、WebSocket 和前端展示通过真实启动后端与 CityFlow 服务进行联调。

### 6.1 测试目标

本次系统测试主要验证以下目标：

1. 验证用户认证、首页数据展示、仿真会话创建、实时帧推送等基础功能是否可用。
2. 验证 Fixed-Time、MaxPressure、Traffic-R 等策略模块是否能进入统一调度流程。
3. 验证安全约束层是否能够拦截非法相位、冲突相位和不满足最小绿灯时间的决策。
4. 验证数据库是否只保存决策、推理、安全、fallback、告警、应急、Agent 调用等复盘数据，不默认持久化高频仿真快照。
5. 验证 Agent 是否能够根据用户问题自动选择工具，并将工具结果与知识库检索切片交给大模型生成最终回答。
6. 验证 Agent 输出是否面向用户展示最终结论，不泄露内部 JSON 字段、工具调用过程和大段原始数据。
7. 验证系统在常规规模路网下的实时性、稳定性和可扩展性是否满足项目演示与实验分析需求。

| 输入决策 | 安全层结果 | 说明 |
|---|---|---|

## 6.7 测试中发现的问题与处理

| 编号 | 问题 | 原因分析 | 处理方式 | 当前状态 |
|---|---|---|---|---|
| P-001 | Traffic-R 健康检查显示 DOWN | 本地隧道 `127.0.0.1:16008` 未连接或连接被拒绝 | 在健康检查中暴露 Traffic-R 和 tunnel 状态，便于排查 | 已处理，模型调用需继续联调 |
| P-002 | Agent 曾输出 JSON 字段给前端 | 模型输出和工具证据没有做最终展示层过滤 | 增加响应组装与格式要求，只向用户展示最终回答 | 已处理 |
| P-003 | 实时查询曾默认读取数据库快照 | 早期设计将仿真快照作为查询来源 | 改为读取 LiveSimulationStateService 内存最新帧 | 已处理 |
| P-004 | 高频仿真帧落库成本过高 | 帧数据频率高，关系数据库不适合保存全量实时帧 | 数据库只保存复盘和分析数据，实时帧保存在内存缓存 | 已处理 |
| P-005 | 安全层可能阻断 Traffic-R 决策 | 决策切换不满足最小绿灯时间 | 改为等待满足最小绿灯时间后再应用策略，不直接危险下发 | 已处理 |
| P-006 | 知识库返回为语义切片而非最终回答 | 百炼 Retrieve API 的定位是检索，不是生成 | 将切片作为证据转交 LLM 生成最终回答 | 已处理 |

## 6.8 测试结论

通过本轮系统测试可以看出，交通信号控制系统已经具备完整的基础运行能力。系统能够创建并运行 CityFlow 仿真会话，前端能够接收后端推送的真实仿真帧，控制策略能够进入统一调度流程，安全约束层能够在信号下发前进行保护，数据库能够保存决策、推理、安全、fallback、应急和 Agent 调用等复盘数据。

Agent 模块已经具备基础可用能力，能够根据用户问题调用实时状态工具、决策追踪工具、系统健康工具、应急调度工具和知识库检索工具。实时类工具读取内存仿真状态，历史复盘类工具读取数据库，符合系统对实时性和数据持久化边界的要求。

当前系统仍需在后续阶段继续补充大规模路网测试、Traffic-R 云端模型稳定调用测试、策略效果对比实验和更完整的性能测试。但从项目验收角度看，系统已经完成主要功能闭环，具备进行演示、实验和最终文档撰写的基础。

## 一  需求调研解决问题<font style="color:#0C68CA;">（LZT）</font>
+ **谁在用？**  
交警指挥中心、交通管理部门、急救/消防调度中心、运维人员、领导驾驶舱、算法工程师等。
+ **解决什么业务问题？**  
拥堵、固定配时不灵活、多路口缺乏协同、应急车辆通行慢、调度决策缺乏依据。
+ **系统控制到什么程度？**  
只是仿真演示？还是能接真实信号机？是建议控制，还是自动下发控制？
+ **需要哪些数据？数据是否能拿到？**  
路口流量、车速、排队长度、信号灯状态、摄像头/雷达/地磁数据、GPS、应急车辆位置、路网拓扑等。
+ **AI 控制的效果如何衡量？**  
平均延误、排队长度、通行效率、绿波通过率、应急车辆通行时间缩短比例等。
+ **系统上线有什么安全边界？**  
信号控制不能出错，必须有人工接管、策略回退、权限控制、日志审计和异常保护。

## 二 需求调研方法
### 1. 资料调研<font style="color:#0C68CA;">（WKY）</font>
先收集已有资料，不要直接空手访谈。

需要收集：

+ 现有信号灯配时方案:

**周期时长方案**：基于交通流量和时间占有率建立“交通强度模型”，针对2~5个关键相位交叉口优化周期时长，并包含协调控制及双周期判断。

**绿信比方案**：基于关键相位判断，采用并修正“等/不等饱和度”原则，优化有无搭接相位下的绿灯时间合理分配。

**相位差方案**：改进数解法以实现流量不平衡下的双向绿波及双周期子区计算；并应用结合法优化区域控制。

**感应控制方案**：首创可变单位绿灯延长时间的优化模型。

+ 高峰期交通流量数据:

 依据《2024 年度中国主要城市交通分析报告》，国内一线城市主干路早高峰小时断面流量约 3000-4500pcu（标准小汽车），次干路约 1500-2500pcu。高峰时段集中在工作日早 7:30-8:30、晚 17:30-18:30，核心通勤走廊、商圈周边流量较平峰上涨 40% 以上，数据多来自城市交警卡口固定监测统计。  

+ 历史拥堵路段与拥堵时段:

 据历年高德城市交通健康报告，国内城市常发拥堵点位集中在跨江 / 跨河通道、环线进出口、三甲医院及重点中小学周边。工作日拥堵时段为早 7:00-9:00、晚 17:00-19:30，周五晚高峰延长至 20:30；常发拥堵路段日均拥堵时长 2-3.5 小时，节假日景区周边拥堵时长翻倍。  

+ 事故、施工、封路等事件数据:

 根据公安部交管局公开年报，城市道路早晚高峰剐蹭类事故占总量 60% 以上，单起事故平均处置时长 15-30 分钟，易引发次生拥堵。占道施工占道路拥堵诱因的 25% 左右，市政施工多安排在夜间 22:00 - 次日 6:00，大范围封路需提前 72 小时通过交管官方渠道公示。  

+ 应急车辆调度流程

 依据《道路交通安全法》及各地交管应急处置规范，标准流程为：指挥中心接警→定位事发点位→调度就近清障 / 救护 / 消防车辆→同步触发沿线信号绿波优先→联动辖区交警现场疏导→处置完成后恢复通行。核心城区应急响应时效要求≤5 分钟，实行 “先调度、后报备” 机制。  

+ 信号机设备型号和通信协议

 国内主流信号机品牌为海信、海康威视、大华、上海电科，代表型号如海信 SCATS 系列、海康 DS-TPM 系列。通信遵循国标 GB/T 25280-2016，通用底层协议为 TCP/IP、RS485，跨系统对接采用 NTCIP 国家交通通信协议，多数设备支持专网光纤 + 4G 双链路备份。  

+ 现有交通平台接口文档

 国内城市交通综合管理平台普遍采用 RESTful API 架构，开放路况、流量、交通事件、设备状态四类核心接口。鉴权采用 AK/SK+Token 机制，返回格式为 JSON，调用频率依权限分级。公开接口规范可在各地政务数据开放平台查阅，内部业务接口需向交管信息科提交申请获取完整文档。  

+ 管理部门考核指标

<font style="background-color:rgba(0, 0, 0, 0);">依据公安部《城市道路交通管理评价指标体系》，核心考核项包括：高峰时段平均车速、常发拥堵路段数量、交通事故万车死亡率、信号机联网率、应急处置响应时长、群众交通满意度。地方额外增加施工占道管控率、绿波协调路段占比等，季度通报、年度总</font><font style="color:rgba(255, 255, 255, 0.9);background-color:rgba(0, 0, 0, 0);">评，结果纳入部门绩效考核。</font>

### 2. 场景调研<font style="color:#0C68CA;">（HCJ）</font>
#### 调研结论概述
本项目的核心需求应围绕真实交通场景展开，而不是单纯围绕技术功能展开。调研后可以明确：城市路网信号控制的主要问题集中在**高峰拥堵、多路口联动失效、应急车辆通行受阻、突发事件扰动、调度展示与决策不直观**五类场景。

#### 场景一：普通早晚高峰拥堵
##### 调研发现的问题
早晚高峰具有明显的潮汐车流特征，部分主干道、快速路衔接路口、商业区和居住区出入口容易出现持续拥堵。传统固定配时方案难以及时适应车流变化，常出现“车多方向绿灯不足、车少方向绿灯浪费”的问题。

当某一方向排队过长时，拥堵可能向上游路口扩散，进一步造成区域性堵塞。严重情况下需要人工介入调整信号灯，调度效率较低。

系统需求设计

+ 系统应实时展示各路口拥堵状态、排队长度、等待时间和通行量。
+ 系统应识别早晚高峰期间的重点拥堵路口和拥堵持续时间。
+ 系统应支持对固定配时方案进行动态调整。
+ 系统应提供人工干预入口，允许调度人员在特殊情况下接管或切换控制方案。
+ 系统应记录优化前后的效果，用于评估平均等待时间、排队长度、通行效率等指标变化。

---

#### 场景二：多路口协调控制
##### 调研发现的问题
单个路口优化并不一定能改善整体路网运行。现实中常见的问题是：上游路口放行过多，但下游路口已经拥堵，导致车辆进入后无法及时疏散，形成“上游放行、下游堵死”的连锁问题。

在主干道场景中，需要保障车辆连续通行；在中心城区网格路网中，则需要兼顾多个方向的交通压力，不能只保障某一条道路。

##### 系统需求设计
+ 系统应支持将多个路口划分为干线走廊或区域协调组。
+ 系统应展示相邻路口之间的交通影响关系。
+ 系统应识别下游路口是否存在排队溢出风险。
+ 系统应支持干线绿波协调，减少车辆在连续路口的停车次数。
+ 系统应支持区域协调控制，避免局部优化导致整体拥堵。
+ 系统应展示多路口协调前后的通行效率对比。

---

#### 场景三：应急车辆通行
##### 调研发现的问题
救护车、消防车等应急车辆在执行任务时，需要尽可能减少红灯等待和拥堵阻碍。但应急绿波不能简单理解为“一路全绿”，否则会严重影响普通车辆通行，并可能破坏原有信号协调关系。

应急车辆通行过程中还存在车辆位置上报、提前感知、信号切换安全、任务结束恢复、多辆应急车冲突等问题。

##### 系统需求设计
+ 系统应支持创建应急任务，包括车辆类型、起点、终点、任务优先级。
+ 系统应支持应急车辆位置上报或模拟位置更新。
+ 系统应为应急车辆生成推荐通行路线。
+ 系统应预测应急车辆到达沿线路口的时间。
+ 系统应在车辆接近路口前提前调整信号灯，为其形成动态绿波通道。
+ 系统应显示当前受影响的路口、信号状态和应急车辆位置。
+ 应急车辆通过后，系统应逐步恢复正常控制状态。
+ 多辆应急车辆同时出现时，系统应支持按车辆类型、任务紧急程度和到达时间进行优先级排序。
+ 系统应生成应急任务复盘报告，包括通行时间、减少停车次数、影响路口和普通交通影响范围。

---

#### 场景四：交通事件扰动
##### 调研发现的问题
交通事故、道路施工、临时管制和恶劣天气会导致局部道路通行能力下降，使原有信号方案失效。此类事件通常具有突发性，若处理不及时，拥堵会向周边路网扩散。

完全自动处置交通事件存在安全风险，现实中更合理的方式是系统先识别异常、评估影响，再向调度人员推荐处置方案，由人工确认后执行。

##### 系统需求设计
+ 系统应支持录入事故、施工、管制、天气等交通事件。
+ 系统应识别异常拥堵，例如非高峰时段排队突然增长、车速明显下降等。
+ 系统应评估事件影响范围，包括受影响路段、周边路口和可能扩散方向。
+ 系统应向调度人员推荐信号调整方案。
+ 系统应支持人工确认后执行方案，避免系统自动误判带来的风险。
+ 系统应持续监控事件处置过程，并记录事件开始时间、处置措施、恢复时间和影响结果。
+ 系统应生成事件复盘记录，为后续优化提供依据。

---

#### 场景五：领导与运维大屏展示
##### 调研发现的问题
数据大屏既要服务领导汇报，也要服务日常运维。领导更关注整体运行效果和项目成果，例如拥堵是否下降、通行效率是否提升、应急车辆是否更快通过；运维人员则更关注实时状态、告警信息、路口细节和处置过程。

如果大屏只展示地图和动画，难以体现系统价值；如果只展示数据指标，又缺乏直观性。因此需要同时兼顾宏观展示和微观运维。

##### 系统需求设计
+ 系统应提供区域路网数字孪生展示，显示路口、道路、车辆流动和信号灯状态。
+ 系统应提供拥堵热力图，直观展示拥堵严重区域。
+ 系统应展示核心指标卡片，包括平均等待时间、排队长度、通行量、拥堵指数、应急车辆通行时间等。
+ 系统应展示固定配时与优化方案的效果对比。
+ 系统应展示应急车辆路径、沿线绿波路口和任务进度。
+ 系统应提供事件告警栏，展示事故、施工、异常拥堵等信息。
+ 系统应区分“汇报展示视图”和“运维监控视图”：
    - 汇报视图突出整体效果、优化成果和应急演示。
    - 运维视图突出实时状态、路口详情、告警处置和操作记录。

### 5. 智能体辅助决策 + RAG 知识库问答<font style="color:#0C68CA;">（HCJ）</font>
**一、调研结论：智能体的系统边界**

真实交通信号系统属于安全关键系统，信号配时涉及最小绿灯、黄灯、全红、行人清空、冲突相位等安全约束。公开资料中，交通信号控制通常由控制器、检测器、配时方案和交通管理中心共同完成；自适应信号控制会根据实际交通需求调整配时，但仍受信号控制规则约束。应急车辆优先系统也不是任意切灯，而是在车辆接近时通过预设机制抢占或优先放行，并保留必要的安全转换时间。

因此，本项目智能体边界建议为：

```plain
允许：查询、解释、统计、建议、生成报告、辅助调度
谨慎允许：生成待确认的调度方案
不允许：绕过安全校验直接下发信号灯控制指令
```

#### 二、用户会问哪些问题？
用户主要分为三类：运维调度员、项目展示人员、系统管理员。

#### 1. 实时路况查询类
用户希望快速知道当前路网运行状态。

典型问题：

当前哪个路口最拥堵？

当前区域平均等待时间是多少？

哪些路口排队长度超过阈值？

哪条道路出现了异常拥堵？

A3 路口当前是什么信号相位？

当前系统运行的是固定配时还是自适应配时？

系统需求：

支持自然语言查询实时路网状态。

返回结果应包含路口名称、拥堵等级、关键指标和时间戳。

查询结果应能跳转或定位到大屏中的对应路口。

#### 2. 拥堵原因解释类
用户不只想知道“哪里堵”，还想知道“为什么堵”。

典型问题：

为什么 XX 路口排队严重？

A3 路口为什么绿灯延长了？

为什么上游车辆没有继续放行？

为什么某一路口没有被纳入绿波控制？

当前拥堵是高峰流量导致，还是下游溢出导致？

系统需求：

智能体应调用拥堵分析、历史状态和策略记录。

回答应解释拥堵原因，而不是只返回指标。

对算法决策应给出可理解说明，例如“下游排队过长，因此限制上游放行”。

#### 3. 应急车辆任务类
用户希望了解应急车辆通行过程和绿波效果。

典型问题：

应急车辆还有多久到达医院？

救护车还有多久到达 B2 路口？

本次应急绿波影响了哪些路口？

当前哪些路口处于应急优先状态？

应急绿波结束后多久恢复正常？

多辆应急车辆同时出现时，谁优先？

系统需求：

支持查询应急车辆位置、路线、预计到达时间。

支持展示沿线路口的绿波状态。

支持说明优先级排序依据。

支持生成应急任务复盘。

#### 4. 调度建议类
用户希望智能体帮助形成方案，但不希望它越权执行。

典型问题：

请生成当前区域信号优化建议。

当前是否需要启动干线绿波？

如果关闭应急绿波，会有什么影响？

针对 A3-A5 连续拥堵，有什么处置建议？

当前事故影响哪些路口，应该如何调整信号？

系统需求：

智能体可以生成调度建议。

建议应包含原因、涉及路口、预计效果和风险。

建议不应自动执行，应进入“待确认方案”状态。

调度员确认后，再由控制模块执行。

#### 5. 历史数据与复盘类
用户需要复盘系统效果和事件处置。

典型问题：

查询某路口的历史配时方案。

昨天晚高峰哪个路口最堵？

固定配时和自适应配时效果差多少？

这次应急绿波节省了多少时间？

今天发生了哪些异常交通事件？

生成今天的交通运行报告。

系统需求：

支持查询历史配时、历史拥堵、历史事件。

支持对比不同策略的效果。

支持自动生成日报、事件报告和优化效果说明。

#### 6. 规范知识问答类
用户需要查询交通管理规范、操作手册和系统规则。

典型问题：

根据交通管理规范，最小绿灯时间是多少？

应急绿波切换前是否需要黄灯和全红？

信号控制中什么是周期、绿信比、相位差？

什么情况下可以人工接管信号控制？

应急车辆优先和普通绿波有什么区别？

系统需求：

智能体应基于知识库回答。

回答必须显示引用来源或知识依据。

如果知识库没有依据，应明确说明“未检索到依据”，不能编造。

#### 三、知识库应包括什么内容？
知识库不应只放项目介绍，而应服务于“可解释、可追溯、可运维”。

1. 交通信号基础知识

包括：

+ 信号周期
+ 相位
+ 绿信比
+ 相位差
+ 黄灯时间
+ 全红时间
+ 最小绿灯时间
+ 行人清空时间
+ 干线绿波
+ 区域协调

用途：

+ 支持运维人员问答。
+ 支持智能体解释信号控制术语。

2. 交通管理规范与安全约束

包括：

+ 信号灯相位冲突规则
+ 最小绿灯约束
+ 黄灯与全红转换规则
+ 行人过街安全要求
+ 应急车辆优先控制原则
+ 人工接管和回退规则

用途：

+ 限制智能体回答边界。
+ 支持答案溯源。
+ 防止生成不安全建议。

3. 系统操作手册

包括：

如何查看路口状态

如何创建应急任务

如何查看绿波路线

如何确认调度建议

如何切换控制策略

如何回退固定配时

如何导出事件报告

用途：

支持运维问答。

支持新用户快速理解系统操作。

4. 项目业务规则

包括：

拥堵等级定义

应急车辆优先级规则

调度建议生成规则

人工确认流程

异常事件分类

大屏指标解释

用途：

保证智能体回答与本项目一致。

避免回答偏离系统实际能力。

5. 历史运行数据与事件记录

包括：

历史配时方案

历史拥堵记录

应急绿波任务记录

策略执行日志

调度员确认记录

优化前后指标

用途：

支持复盘类问题。

支持效果对比类问题。

支持生成报告。

#### 四、智能体是否只是问答？
不建议只做问答。只做问答会显得价值偏弱。

更合理的定位是：

智能体 = 自然语言入口 + 工具调用编排 + 知识库问答 + 调度建议生成 + 结果解释。

也就是说，智能体既能问答，也能调用系统工具，但调用范围要受控。

#### 可调用工具
| 工具能力 | 是否允许 | 说明 |
| --- | --- | --- |
| 查询实时路况 | 允许 | 获取路口状态、排队长度、等待时间 |
| 查询历史数据 | 允许 | 查询配时、拥堵、事件、策略记录 |
| 拥堵分析 | 允许 | 判断拥堵等级和疑似原因 |
| 应急车辆 ETA 查询 | 允许 | 查询车辆到达时间和路线进度 |
| 指标统计 | 允许 | 统计等待时间、通行量、节省时间 |
| 策略效果对比 | 允许 | 对比固定配时、自适应配时、绿波效果 |
| 生成调度建议 | 允许，但需人工确认 | 形成待确认方案 |
| 下发信号控制 | 第一阶段不允许 | 防止越权和安全风险 |


---

#### 五、是否允许智能体生成调度建议？
建议允许，但必须限制为“建议”，不能自动执行。

调度建议应包含

+ 当前问题描述
+ 涉及路口或路段
+ 建议调整方向
+ 调整原因
+ 预计改善指标
+ 可能影响范围
+ 风险提示
+ 是否需要人工确认

示例：

建议对 A3-A5 干线启动协调控制。  
原因：A3 北向排队长度持续超过阈值，A4 下游仍有可用空间，A5 当前未出现溢出风险。  
预计效果：可减少主方向停车次数，并降低平均等待时间。  
风险：东西向支路等待时间可能短时增加。  
建议状态：待调度员确认。

---

#### 六、是否允许智能体下发控制指令？
第一阶段不建议允许。

原因：

+ 信号灯控制是安全关键操作。
+ 智能体可能存在误解、幻觉或上下文遗漏。
+ 真实信号控制必须满足严格安全约束。
+ 调度策略需要可追溯、可回退、可人工确认。

##### 第一阶段边界
```plain
智能体可以生成建议
智能体可以解释方案
智能体可以创建待确认任务
智能体不能直接改变信号灯
```

##### 第二阶段可扩展
如果后续系统成熟，可以允许智能体在低风险场景下调用“受限控制接口”，但必须满足：

+ 用户具有权限
+ 控制范围白名单
+ 方案通过安全校验
+ 操作需要二次确认
+ 保存完整日志
+ 支持一键回退

---
